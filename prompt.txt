# 🚀 Prompt: Build “NebulaCB” — Couchbase Migration & Upgrade Mission Control

Create a production-grade distributed system called **NebulaCB** — a modern Couchbase migration, upgrade validation, and resilience testing platform designed for multi-region, high-ingestion environments.

## 🎯 Objective

NebulaCB must simulate, validate, and observe Couchbase cluster upgrades (e.g., 7.2.2 → 7.6.x) under **real-world load conditions**, ensuring:

* Zero data loss
* Continuous ingestion during upgrade
* XDCR replication integrity
* Automatic detection of replication gaps or inconsistencies
* Full observability with a **mission-control cockpit dashboard**

---

## 🧱 Core Architecture

Design NebulaCB as a cloud-native system with:

* Golang backend (high concurrency, low overhead)
* Kubernetes-native deployment (Helm + CRDs)
* Modular services:

### 1. Data Storm Generator 🌩️

A high-throughput data generator that:

* Writes millions of documents/sec (configurable)
* Supports:

  * KV operations (insert/update/delete)
  * Large documents (1KB → 1MB)
  * Hot/cold key distribution
* Simulates:

  * Bursty traffic
  * Sustained high load
  * Multi-region writes

Must include:

* Sequence IDs
* Timestamps
* Region tags
* Checksum/hash per document

---

### 2. Upgrade Orchestrator ⚙️

Automates Couchbase upgrades:

* Helm-based upgrade execution
* Node-by-node rolling upgrade tracking
* Operator-aware (Couchbase Autonomous Operator)
* Ability to:

  * Pause/resume upgrade
  * Inject failures (kill pod, delay rebalance)
  * Control upgrade speed

---

### 3. XDCR Chaos & Validation Engine 🔁

Based on real-world behaviour:

* Detect:

  * Topology changes
  * VBucket rebalance events
  * XDCR pipeline pauses (~5 min GOXDCR delay)
* Track:

  * Replication lag
  * Mutation queue depth
  * Pipeline restarts

Must simulate:

* Continuous replication during upgrade
* Cross-version replication (7.2 → 7.6)

---

### 4. Data Integrity Validator 🔍

Critical component:

* Continuously compare:

  * Source vs Target clusters
* Validate:

  * Document count
  * Key presence
  * Hash consistency
  * Sequence continuity

If data loss occurs:

* Detect missing keys
* Detect stale updates
* Detect duplication

---
